## Recurring patterns observed

### P1 — Brittle literal-token acceptance criteria

The single dominant signal across three phases. Phase 2 named it as F11
(deferred Improvement). Phase 3 reconfirmed it concretely: 4 of 24 AC failed
mechanically on literal-token mismatch (AC-1, AC-2, AC-5, AC-18) and required
Route-2 revision. Phase 4 critique surfaced a third generation: AC-17's
positive-signal leg used a loose alternation (`PASS|critique`) that would also
match `phase-4 critique missing` — same family (literal token under a
permissive alternation defeats the positive-signal intent).

Lesson: prose-anchored greps couple AC to artifact wording. The AC then
mechanically fails when the artifact was written correctly but using
synonyms or section names the AC author did not anticipate. Two of three
phases needed a remediation round purely to align literal tokens.

### P2 — Anti-leak compound-pipeline blind spots

Phase 3 F12 documented that the original AC-15 pipeline filtered through a
final `grep -E "\bBW-[0-9A-Z]+\b"`, which silently dropped any leak that did
not happen to contain a ticket token. Phase 4 split the check into two
independent legs (keyword leg + ticket leg, both must return zero). The
underlying lesson: compound pipes that chain include + exclude + final filter
hide failure conditions.

### P3 — Exclusion-list drift surface

Phase 3 F13, F14 and F15 all sit in the same family: an exclusion regex was
narrow enough today, but unanchored or operator-drift-prone tomorrow. F15 made
the cultural rule explicit ("fix the artifact, don't widen the exclusion").
Phase 4 F13 carried the same pattern to the vault side: vault AC-12 threshold
of "≥2 of 5 alternatives" allowed a vault file to silently drop the
"every 3 closed tickets" trigger and still pass.

### P4 — Validator UX vs the "no validator changes" constraint

Phase 4 AC-17 PARTIAL: a positive-signal AC ("validator emits a phase-N PASS
marker") cannot be satisfied without patching the validator, but the phase's
own Constraints forbid validator changes. Self-imposed tension. The retro is
the right place to reconcile.

### P5 — Spec-critic cadence: two passes is the norm, not the exception

Phase 2, Phase 3, and Phase 4 all required a `SPEC_BLOCKED → SPEC_CRITIQUED`
revision round. The gate is doing its job (it caught real ambiguity each
time), but cadence-wise this is a stable feature, not a transitional one.

### P6 — Route-1 vs Route-2 remediation decision is ad-hoc

Phase 2 chose Route 1 (edit content to match the AC literal). Phase 3 chose
Route 2 (revise the AC to match the delivered content). Both were correct in
context, but the choice was made ad-hoc each time. The criterion is
implicit ("did the content drift, or was the AC wrong?") and would benefit
from being written down.

### P7 — Discovery artifact for Critical-lane sensitive material

Phase 3 security INFO #2 flagged a future concern: discovery is "purely
optional" by design, but if it ever carries sensitive technical alternatives
(e.g. cryptographic primitive choices) for a Critical-lane ticket, it bypasses
spec-critic / security-reviewer by construction. Today the trust boundary is
the PRD; for Critical lane this may need to extend.

### P8 — Docs-sync gate is manual by design

Phase 4 Phase-1 idea decision and Q1 deliberately kept docs-sync manual.
Phase 4 security INFO-4 explicitly accepted the residual risk. Acceptable for
v3.2; revisit only if drift recurs in future retrospectives.

## Friction points reported mid-flight

- **Self-application bootstrap paradox** — Phase 2 had to install the
  `SPEC_CRITIQUED` gate while being the first phase to run under it. The
  bootstrap-exempt INFO line was the workable compromise but introduced an
  exemption surface that every later phase has to keep scoped.
- **Orchestrator action ordering** — Phase 3 AC-13b (memory pointer check)
  was deferred to Batch 7 because memory cleanup had to happen *after* repo
  commits landed (otherwise the pointers would point at unmerged paths).
  The dependency was not obvious from the AC table alone.
- **Manual handoff chain** — Critical lane requires
  reviewer → security-reviewer → qa. No auto-handoff exists; orchestrator
  drives each step. This is by design but the cost is real.
- **PRD wording inertia** — Phase 4 Q3 dropped the 30-day calendar trigger
  but Negative scenario N5 hinted the trigger thresholds may themselves
  need broadening through the retrospective. Phase 4 F15 noted that the
  "retrospective is the canonical channel for adjusting its own triggers"
  was not explicit. (Addressed below.)

## Candidate changes for AIDD v3.3

Each candidate is sized for a separate Professional- or Critical-lane phase
under a future meta-ticket (working name: **BW-META-002**, to be opened in
a separate session per the user's instruction). None are implemented here —
the retrospective surfaces them; the meta-ticket scopes them.

### C1 — Verifiable AC: written guidance against brittle literal-token greps

Add a sub-section to `docs/project/workflow.md § Verifiable AC` (and the vault
mirror in `Methodology/Artifact Contracts.md`) documenting:

- Prefer **count-based** AC over literal-token AC for prose surfaces
  (e.g. "≥5 `^## ` headers" rather than "section named `## Vision`").
- Reserve literal-token greps for **code, regex anchors, and validator
  source** — i.e. surfaces where the literal *is* the contract.
- When a literal token AC is unavoidable for prose, pair it with a
  semantic alternative (`comm -23 expected actual` returns empty) so the
  failure message names the missing piece.
- Document the anti-pattern explicitly with the Phase 3 AC-1/AC-2/AC-5/AC-18
  case as the example.

Drives: P1, P3 (operator-drift), F11 Phase 2/3, F14 Phase 4, F16 Phase 3.

### C2 — Canonicalise the two-leg anti-leak pattern

Promote the Phase-4 split (keyword leg + ticket leg, evaluated under
`set -e`, each returning zero independently) from "this PRD did it" to "this
is the canonical shape" in `docs/project/workflow.md`. Forbid the
single-pipe compound form in new PRDs.

Drives: P2, F11 Phase 3, F12 Phase 3 (negative-regex blind spot is the
same family).

### C3 — Exclusion-pattern hygiene rule

One paragraph under `## Verifiable AC` or the new Verifiable AC sub-section
(see C1):

> When an AC includes an exclusion (`grep -v ...`), anchor filename patterns
> to a directory segment (e.g. `/phase/phase-3.md`, not `/phase-3.md`) and
> add the operator-drift caveat verbatim: "exclusion list is exhaustive; if
> a stale reference surfaces under a non-excluded subdir, fix the artifact
> rather than widen the exclusion."

Drives: P3, F14 Phase 3, F15 Phase 3.

### C4 — Validator UX: allow emit-only positive-signal lines

Reconcile the AC-17 PARTIAL leg. Two routes; the retrospective recommends
the first:

a. Patch `aidd_validate.sh` to emit a per-phase `INFO: phase-N OK` line on
   success, and classify the patch as Trivial-lane (no behaviour change,
   no FAIL count change, no exit-code change). Wording: "emit-only UX
   patches are Trivial-lane eligible when no gate semantics change".
b. Relax the AC wording so positive-signal claims do not depend on
   validator output (rely on exit code alone).

Drives: P4, Phase 4 F14 (loose alternation symptom), Phase 4
Outstanding §AC-17.

### C5 — Vault/project AC symmetry threshold

When a project-side AC and a vault-side AC test the same concept set, the
vault threshold must be `≥N` where N equals the number of distinct concepts
named in the project AC, not `≥2 of 5 alternatives`. Document the rule and
update the v3.3 PRD template to call it out.

Drives: F13 Phase 4, P3.

### C6 — Discovery artifact for Critical-lane sensitive material

Add a single Critical-lane sub-rule in `docs/project/workflow.md §
Discovery artifact`:

> Discovery artifacts for Critical-lane tickets that touch wallet / keys /
> crypto / signing / storage / auth surfaces must be routed through
> `security-reviewer` even though they are otherwise optional.

Drives: P7, Phase 3 security INFO-2.

### C7 — Document the Route-1 vs Route-2 remediation decision

Add to `docs/project/workflow.md § Verifiable AC` (or the new sub-section):

> When a `command:` AC fails, choose the remediation route by asking
> "did the implementation drift, or did the AC predict the wrong shape?"
> Route 1 (edit content to match AC) if the AC describes the intended
> shape and the artifact missed it. Route 2 (revise the AC) if the
> artifact is correct and the AC was over-specified. Record the route
> chosen in the phase summary's `## Critique handling` table.

Drives: P6, Phase 2/3 transparency-note inconsistency.

### C8 — Retrospective is the canonical trigger-modification channel

Add one explicit sentence to `docs/project/workflow.md § Retrospective
triggers`:

> The retrospective process is itself the canonical channel for adjusting
> its own triggers. Initial threshold values (1 Critical-lane closure,
> cumulative ≥ 2 Blocking findings, every 3 Professional/Critical closures)
> are intentional defaults, not invariants; the v3.3 retrospective may
> revise them.

Drives: F15 Phase 4, friction note above.

### C9 — Trivial-lane edit: literal-anchor inline comment in validator

`aidd_validate.sh:387` carries a literal `BW-0018` anchor in two places. Add
an inline comment block warning future maintainers "literal `BW-0018`; do
not parametrize". This is the smallest item in the retrospective: pure
Trivial-lane work, single-file, no behaviour change.

Drives: Phase 3 security INFO-1.

### C10 — Spec-critic cadence is "two passes by default"

No code change. Document in `docs/project/workflow.md § Spec-critic` that
two passes (initial SPEC_BLOCKED → revision → SPEC_CRITIQUED) is the
expected cadence, not a sign of low-quality PRD authoring. Adjust phase
planning effort estimates accordingly.

Drives: P5, observed cadence across all three gated phases.
